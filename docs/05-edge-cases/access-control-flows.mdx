# GMC Education Portal — School-Controlled Access Flows

**Version:** 1.0
**Last Updated:** May 2026
**Owner:** GMC Product Team

---

## Core Principle

> No user can create their own account. Every account is created from the top down — GMC creates schools, schools create principals, principals create teachers, teachers create students. Parents get access only through their registered mobile number linked to their child.

This is not just a design choice. It is a security requirement for government student data.

---

## Access Hierarchy

```
GMC Admin
    ↓ creates
School (with Principal)
    ↓ creates
Teachers
    ↓ creates / imports
Students
    ↓ linked by mobile number
Parents
```

No one enters this chain from the side. No self-registration. No invite links that anyone can use. No open signup page.

# SCENARIO 1 — New School Year Setup

**When it happens:** Every June before the academic year starts.

**Who does it:** GMC Admin → Principal → Teachers

```
STEP 1 — GMC Admin runs New Year Migration

GMC Admin Portal → Settings → New Academic Year

System asks:
"Start new year 2026-27?
This will:
- Archive all 2025-26 student data (read-only)
- Promote passing students to next class
- Remove graduated Class 10 students
- Keep all teacher accounts active
Confirm?"

Admin types: "2026-27" to confirm

↓

System processes:
- Class 6 students → Class 7
- Class 7 students → Class 8
- Class 8 students → Class 9
- Class 9 students → Class 10
- Class 10 students → Archived (graduated)
- Failed students → Held back (flagged for principal review)

↓

STEP 2 — GMC Admin imports new Class 6 admissions

New students CSV uploaded:
Name, Roll Number, Class, Section, Parent Mobile

System creates accounts for all new Class 6 students
Login SMS sent to parent numbers:
"Your child [Name] has been enrolled at [School].
Student login — Roll: [XXX] PIN: [XXXX]
Download GMC Shiksha: [link]"

↓

STEP 3 — Principal reviews held-back students

Principal Portal → Students → Held Back (8 students)

For each student:
- Confirm held back → student stays in same class
- Override to promote → student moves up with note

↓

STEP 4 — Teachers assigned to new classes

Principal → Teachers → Assign Classes
Rameshbhai Patel → Maths → Class 8A, 8B, 9A
Sunita Ben Shah → Science → Class 7A, 7B, 8A

↓

STEP 5 — New academic year is live

All students can log in with new class assignments
Timetables updated
Previous year data archived but accessible to principal

✅ School year ready — 0 manual account creation needed
```

## Summary — Why School-Controlled Access Works

| Risk | How It Is Prevented |
|---|---|
| Random person creating fake student account | No signup page exists anywhere |
| Student accessing another student's data | Account lock after 5 attempts, brute force detection |
| Parent seeing wrong child's data | Parent linked only to registered child's roll number |
| Teacher accessing another school's data | Role-based access — school-scoped at API level |
| Data manipulation without accountability | Every edit requires reason, approver, audit log entry |
| Account persisting after student leaves | Principal deactivates — immediate effect |
| Teacher transfer data leak | Transfer process preserves data, revokes cross-school access |
| Inspector finding discrepancies | Full audit trail — every action logged permanently |

---

*These flows represent the complete access lifecycle of the GMC Education Portal. Every new feature built must fit within this hierarchy. If a feature requires bypassing any level of this chain, it must be reviewed by the Tech Lead and PM before implementation.*
